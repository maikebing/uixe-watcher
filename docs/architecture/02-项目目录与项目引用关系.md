# Uixe.Watcher 项目目录与项目引用关系

## 1. 文档目的

本文档用于明确 `Uixe.Watcher` 的推荐目录结构、项目职责、项目引用方向和依赖边界，避免在迁移过程中出现边界混乱、跨层依赖和重复建设。

---

## 3. 项目职责说明

### 3.1 `Uixe.Watcher.Api`

职责：

- 提供 HTTP API
- 承载 Swagger/OpenAPI
- 暴露 `SignalR Hub`
- 处理认证、授权、异常、日志等入口层能力

不应承担：

- 业务核心逻辑
- UI 操作逻辑
- 数据访问细节

### 3.2 `Uixe.Watcher.Application`

职责：

- 承载业务用例编排
- 承载命令和查询
- 暴露应用服务接口
- 协调领域对象与基础设施实现

建议目录：

- `Abstractions`
- `Services`
- `Features`
- `Commands`
- `Queries`

### 3.3 `Uixe.Watcher.Domain`

职责：

- 实体
- 值对象
- 枚举
- 领域规则
- 领域事件

要求：

- 保持纯净
- 不依赖具体 UI、数据库、缓存和外部接口实现

### 3.4 `Uixe.Watcher.Infrastructure`

职责：

- 数据访问
- 缓存实现
- 外部系统调用
- Quartz 任务
- 推送实现
- 媒体访问与解析

建议目录：

- `Persistence`
- `Caching`
- `Integrations`
- `Jobs`
- `Messaging`

### 3.5 `Uixe.Watcher.Contracts`

职责：

- 请求模型
- 响应模型
- DTO
- 共享枚举
- SignalR 消息契约

说明：

- 用于统一对外契约和跨项目共享模型
- 不承载业务逻辑

### 3.6 `Uixe.Watcher.Agent`

职责：

- 本地旧接口兼容
- 请求转发
- 本地通知
- 语音播报
- `VNC` 调起
- `WebView` 承载

### 3.7 `Uixe.Watcher.WinForms.Legacy`

职责：

- 旧界面保留
- 过渡期展示
- 兼容本地交互
- 逐步退出业务中心角色

说明：

- 迁移阶段不直接删除
- 可与新 Web 界面并存和切换

## 4. 推荐引用关系

### 4.1 核心引用方向

推荐引用方向如下：

- `Uixe.Watcher.Api` -> `Uixe.Watcher.Application`
- `Uixe.Watcher.Api` -> `Uixe.Watcher.Contracts`
- `Uixe.Watcher.Application` -> `Uixe.Watcher.Domain`
- `Uixe.Watcher.Application` -> `Uixe.Watcher.Contracts`
- `Uixe.Watcher.Infrastructure` -> `Uixe.Watcher.Application`
- `Uixe.Watcher.Infrastructure` -> `Uixe.Watcher.Domain`
- `Uixe.Watcher.Infrastructure` -> `Uixe.Watcher.Contracts`
- `Uixe.Watcher.Agent` -> `Uixe.Watcher.Contracts`
- `Uixe.Watcher.Agent` -> `Uixe.Watcher.Application`（仅在确有需要时）
- `Uixe.Watcher.WinForms.Legacy` -> `Uixe.Watcher.Contracts`
- `Uixe.Watcher.WinForms.Legacy` -> `Uixe.Watcher.Application`（过渡期允许）

### 4.2 不允许的引用方向

以下引用方向应避免或禁止：

- `Domain` -> `Infrastructure`
- `Domain` -> `Api`
- `Domain` -> `WinForms`
- `Application` -> `WinForms`
- `Application` -> `Api`
- `Contracts` -> `Infrastructure`
- `Contracts` -> `WinForms`
- `Api` -> `WinForms`

## 5. 项目引用边界规则

### 5.1 API 层规则

- 只依赖应用层和契约层
- 不直接引用旧窗体工程
- 不直接依赖 `frmPlaza`、`frmMain` 等类型

### 5.2 Application 层规则

- 定义接口和业务编排
- 不引用具体 UI 工程
- 不依赖具体数据库或缓存实现

### 5.3 Domain 层规则

- 保持最少依赖
- 不关心宿主、网络和界面
- 不承载基础设施细节

### 5.4 Infrastructure 层规则

- 负责实现应用层定义的接口
- 可以依赖数据库、缓存、中间件和第三方库
- 不向上反向持有 UI 逻辑

### 5.5 Legacy WinForms 层规则

- 可在过渡期调用应用层服务
- 不再作为长期业务中心
- 不新增新的核心业务逻辑承载

---

## 6. 当前项目的过渡映射建议

由于当前仓库仍是单体 `WinForms + 进程内 Web` 结构，建议按以下顺序映射：

### 6.1 第一步

从现有项目中先抽离：

- DTO
- 请求响应模型
- 共享枚举
- 业务接口

### 6.2 第二步

从当前 `Controller` 和服务中拆出：

- 车道匹配逻辑
- 事件处理逻辑
- 媒体整理逻辑
- 通知推送逻辑

### 6.3 第三步

保留当前 `WinForms` 入口，但逐步降低其职责：

- 仅负责展示
- 仅负责本地能力
- 不再直接承担业务核心流程

## 7. 建议的测试结构

### 7.1 单元测试

- `Uixe.Watcher.Application.Tests`
- `Uixe.Watcher.Domain.Tests`

### 7.2 集成测试

- `Uixe.Watcher.Infrastructure.Tests`
- `Uixe.Watcher.IntegrationTests`

### 7.3 接口测试

- `Uixe.Watcher.Api.Tests`

说明：

- 首批测试优先覆盖兼容接口和应用服务
- 重要接口在重构前后应具备回归验证样例

---

## 8. 结论

目录结构和项目引用关系不是形式问题，而是迁移成功的基础约束。

在当前项目中，最重要的边界要求是：

- `Controller` 不再依赖窗体
- 业务逻辑迁移到应用层
- 旧 `WinForms` 只做过渡展示与本地能力承载
- 新后端按独立 `ASP.NET Core 10 Web API` 架构演进
