# Uixe.Watcher 总体架构设计

## 1. 文档目的

本文档用于明确 `Uixe.Watcher` 的目标架构、系统边界、项目拆分方式和演进原则，作为后续开发、迁移和协作的统一基线。

本文档面向以下目标：

- 为现有 `WinForms + 进程内 ASP.NET Core` 模式提供可落地的演进路径
- 为后续新建 `Web API`、`Web` 前端和 `Agent` 兼容层提供统一命名与边界
- 避免一次性重写导致的业务中断和联调风险

---

## 2. 当前系统现状

当前系统的核心形态不是标准前后端分离，而是：

- `WinForms` 主程序作为主宿主
- `ASP.NET Core` 以进程内自托管方式运行
- 外部系统调用本机 `9999`
- `Controller` 直接访问 `frmPlaza` 等窗体对象
- API 收到请求后直接驱动桌面界面和本地能力

当前主链路：

`外部系统 -> 本机内嵌 API -> WinForms 窗体`

### 2.1 当前模式的问题

当前模式可以快速集成本地能力，但存在明显限制：

- 业务处理与桌面展示强耦合
- `Controller` 直接依赖 UI 类型，难以测试和复用
- 无法平滑扩展到集中式后端部署
- 不利于多用户访问、历史查询、统一监控和集中运维
- 难以支持标准化 Web 前端

---

## 3. 目标架构

### 3.1 目标主链路

未来标准链路应为：

`外部系统 -> 中心后端 API -> Application Service -> 数据存储/缓存 -> SignalR -> Web 前端`

如果需要保留本地能力，补充兼容链路：

`中心后端 -> Agent -> 本地弹窗/语音/VNC/设备能力`

### 3.2 架构目标

- 后端独立部署
- 前端独立运行
- 本地能力可选保留
- 旧接口可兼容过渡
- 业务处理与界面展示彻底分离
- 支持历史追溯、统计查询、实时推送

### 3.3 架构原则

1. 先兼容，后替换
2. 不直接删除现有内容
3. 现有界面与新 Web 界面允许并存，并可切换
4. 先抽离业务，再迁移展示
5. 先保接口稳定，再推动上游改造

---

## 4. 技术选型

### 4.1 后端

- `ASP.NET Core 10 Web API`
- `SignalR`
- `Quartz`
- `Swagger / OpenAPI`
- `MemoryCache` 或 `Redis`
- `PostgreSQL` 或 `SQL Server`

### 4.2 前端

- `Vue 3`
- `TypeScript`
- `Vite`
- `Pinia`
- `Vue Router`
- `Arco Design Vue`
- `Tailwind CSS`
- `ECharts`
- `SignalR JavaScript Client`

### 4.3 本地兼容层

- `Uixe.Watcher.Agent`
- 本地通知
- 本地语音播报
- `VNC` 调起
- 旧接口兼容转发
- `WebView` 容器

### 4.4 数据存储建议

不建议继续以 `LiteDB` 作为未来中心后端的核心数据库。

原因：

- 更适合本地单机
- 不适合多用户并发访问
- 不适合集中统计查询
- 不适合历史追溯和统一运维

建议演进方向：

- `PostgreSQL`：适合事件型数据和高性价比部署
- `SQL Server`：适合偏微软栈现场环境

## 5. 逻辑分层

### 5.1 API 层

职责：

- 对外提供 HTTP 接口
- 保持旧接口兼容
- 参数绑定与校验
- 请求映射
- 暴露 `SignalR Hub`
- 统一认证、授权、异常处理与日志

边界要求：

- 不直接引用 `frmPlaza`、`frmMain` 等 UI 类型
- 不直接承载业务编排
- 不直接操作窗体控件

### 5.2 Application 层

职责：

- 承载业务用例编排
- 组织领域对象与基础设施协作
- 处理命令与查询
- 输出标准结果模型
- 统一触发实时推送与通知

建议服务：

- `EventIngestionService`
- `LaneMatchingService`
- `MediaResolverService`
- `RealtimePushService`
- `NotificationService`
- `SystemConfigService`

### 5.3 Domain 层

职责：

- 定义实体
- 定义枚举和值对象
- 定义领域规则
- 表达系统核心业务语义

边界要求：

- 不依赖 UI
- 不依赖具体数据库实现
- 不依赖具体网络传输实现

### 5.4 Infrastructure 层

职责：

- 数据访问
- 缓存实现
- 外部系统对接
- 定时任务
- 消息推送实现
- 文件和媒体解析

### 5.5 Contracts 层

职责：

- 对外 DTO
- 请求响应模型
- 共享枚举
- SignalR 消息契约

说明：

- 该层用于减少前后端、兼容层和应用层之间的契约重复定义

## 7. 关键运行模式

### 7.1 兼容并存模式

在迁移阶段允许以下模式长期并存一段时间：

- 旧 `WinForms` 界面继续运行
- 新 Web 界面同步建设
- 用户可在旧界面和新 Web 界面间切换
- 同一套后端服务同时服务旧接口和新接口

这是当前项目的核心约束，不允许在第一阶段直接删除旧界面。

### 7.2 Agent 过渡模式

如果需要平滑过渡，可引入 `Agent` 承担以下职责：

- 监听旧接口
- 将请求转发到中心后端
- 接收中心后端推送
- 调起本地通知、语音和 `VNC`
- 通过 `WebView` 承载新 Web 界面

### 7.3 最终模式

最终目标是：

- 中心后端独立部署
- Web 前端作为主操作界面
- Agent 只保留本地扩展能力
- 旧 `WinForms` 仅作为过渡层或逐步退出

## 8. 核心接口兼容策略

第一阶段需要优先兼容以下入口：

- `LaneController`
- `TrafficEvent([FromBody] TrafficEventPushRequest)`
- `Bulktrans(string, BulklyDto)`
- `bill_info(string, BillInfoDto)`
- `ConfirmEnInfo`

### 8.1 兼容要求

- 尽量保持现有请求契约不变
- 不先要求上游系统改造
- 允许内部新增适配映射层
- 对旧接口返回结构尽量保持稳定

### 8.2 重构要求

- `Controller` 只接请求和返回结果
- UI 操作移出 `Controller`
- 从 `Controller -> Form` 改为 `Controller -> Application Service`
- 标准事件写库、缓存、推送分离处理

## 9. 当前优先改造对象

优先处理以下文件和模块：

1. `Program.cs`
2. `Startup.cs`
3. `Controllers/LaneController.cs`
4. `Services/TrafficEventQueueService.cs`
5. `WinForms/frmMain.cs`
6. `WinForms/frmPlaza.cs`

### 9.1 目标调整方向

`Program.cs`

- 为独立后端启动做准备
- 保留过渡期宿主能力

`Startup.cs`

- 梳理当前注册内容
- 逐步迁移到独立后端项目

`LaneController.cs`

- 去掉直接操作窗体
- 改为调用应用服务

`TrafficEventQueueService.cs`

- 去掉对窗体依赖
- 改为事件队列、推送和通知协作

`frmMain.cs` / `frmPlaza.cs`

- 从业务承载者降级为展示端与本地能力承载者

## 10. 风险与控制点

### 10.1 主要风险

- 旧接口契约被无意破坏
- UI 行为迁移后出现展示差异
- 业务逻辑在窗体中分散，梳理成本高
- 本地能力迁移到 `Agent` 后存在稳定性风险
- 新旧两套界面并存期间状态同步复杂

### 10.2 控制策略

- 先冻结旧接口清单
- 先梳理 DTO 与事件链路
- 先抽象接口，再迁移实现
- 新旧界面并存期间保留切换入口
- 每阶段保留回滚方案
- 重要节点引入集成测试和联调验证

---

## 11. 架构结论

`Uixe.Watcher` 的推荐演进方式不是推翻重做，而是：

- 保持旧系统可运行
- 新建独立后端分层结构
- 逐步把 UI 相关逻辑从 `Controller` 和服务中剥离
- 用 Web 前端承接未来主要交互
- 用 `Agent` 承接本地能力和旧接口兼容

该方案兼顾了技术演进、落地成本和业务连续性，适合当前项目的实际情况。
