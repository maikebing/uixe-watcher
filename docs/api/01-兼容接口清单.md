# Uixe.Watcher 兼容接口清单

## 1. 文档目的

本文档用于整理 `Uixe.Watcher` 当前迁移阶段必须优先兼容的接口、契约约束、改造边界和实施要求，作为第 0 阶段与第 1 阶段的接口基线。

本文档重点面向：

- 后端解耦
- 旧接口兼容
- 上游联调稳定
- 新旧系统并存阶段的接口管控

---

## 2. 接口兼容原则

### 2.1 总原则

- 先兼容，后替换
- 不先要求上游系统改造
- 不直接删除现有接口
- 允许内部实现重构
- 尽量保持请求契约稳定
- 尽量保持返回结构稳定

### 2.2 实施要求

- `Controller` 不再直接操作 `WinForms`
- 从 `Controller -> Form` 改为 `Controller -> Application Service`
- 外部契约不变时，内部允许增加映射层和适配层
- 新旧界面并存期间，接口必须同时服务旧界面与新 Web 前端

---

## 3. 当前优先兼容接口范围

根据当前 `README.md`、迁移规划和现有接口文档，第一阶段优先兼容以下入口：

1. `LaneController`
2. `TrafficEvent([FromBody] TrafficEventPushRequest)`
3. `Bulktrans(string, BulklyDto)`
4. `bill_info(string, BillInfoDto)`
5. `ConfirmEnInfo`

说明：

- 上述接口属于迁移首批保护对象
- 后续新增标准接口时，也不能先破坏这批旧入口

---

## 4. 接口分类说明

### 4.1 车道与站点相关接口

代表入口：

- `LaneController`

职责推测：

- 接收外部车道或站点相关请求
- 触发车道匹配、站点定位和前端展示刷新
- 目前可能直接操作 `frmPlaza` 或相关 UI

迁移要求：

- 保留现有路由和请求方式
- 控制器只做接收、校验和调用应用服务
- 车道匹配、站点定位等逻辑迁移到 `Application` 层
- UI 展示交由旧界面或新 Web 自行消费

### 4.2 交通事件推送接口

代表入口：

- `TrafficEvent([FromBody] TrafficEventPushRequest)`

根据现有 `docs/交通事件管理接口文档.md`，该类接口至少包含以下典型参数：

- `recordId`
- `eventType`
- `LaneNo`
- `capTime`
- `startTime`
- `period`
- `periodByMili`
- `maxQueueLen`
- `imageList`
- `videoList`

现有返回结构示例：

```json
{
  "code": 0,
  "message": "推送成功",
  "timestamp": 1765941141872
}
```

迁移要求：

- 保持 `TrafficEventPushRequest` 请求契约尽量不变
- 保持旧返回结构兼容
- 内部迁移为事件接收、去重、入库、推送分离流程
- 媒体解析逻辑迁移到独立服务

### 4.3 批量业务接口

代表入口：

- `Bulktrans(string, BulklyDto)`

迁移要求：

- 保持原入参与调用方式
- 如需新增标准模型，通过适配层转换
- 不在控制器里直接写窗体逻辑

### 4.4 账单信息接口

代表入口：

- `bill_info(string, BillInfoDto)`

迁移要求：

- 优先保护上游账单传输契约
- 将业务判断与展示逻辑分离
- 输出统一处理结果模型供 UI 使用

### 4.5 确认类接口

代表入口：

- `ConfirmEnInfo`

迁移要求：

- 保持确认相关请求结构稳定
- 将确认处理迁移到应用服务
- 保留旧界面确认能力，同时预留新 Web 界面确认能力

---

## 5. 当前接口契约保护策略

### 5.1 请求保护

第一阶段要求以下内容尽量不变：

- 请求路由
- 请求方法
- 参数名称
- 参数层级结构
- 必填字段语义

### 5.2 返回保护

第一阶段要求以下内容尽量不变：

- 返回字段名
- 返回码语义
- 成功/失败判断方式
- 关键提示文本格式

### 5.3 特殊情况处理

如果内部重构必须导致外部变化：

- 必须先记录影响范围
- 必须先补兼容适配层
- 必须先给出联调说明
- 未明确批准前，不允许直接破坏旧契约

---

## 6. 推荐接口处理流程

迁移后建议统一采用以下处理链路：

`Controller -> Request Mapping -> Application Service -> Domain / Infrastructure -> Result Mapping -> Response`

### 6.1 Controller 层职责

- 接收请求
- 做基础校验
- 做请求映射
- 调用应用服务
- 返回标准结果

### 6.2 Application 层职责

- 编排业务流程
- 调用车道匹配、媒体解析、通知推送等服务
- 输出统一结果对象

### 6.3 Infrastructure 层职责

- 数据库存取
- 缓存
- 文件与媒体访问
- 外部系统调用
- 实时推送实现

## 9. 联调与测试建议

### 9.1 联调要求

- 保存现有请求样例
- 保存现有返回样例
- 对每个关键接口建立最小回归测试样本
- 新旧实现切换时保留日志对比

### 9.2 测试建议

优先覆盖：

- 正常请求
- 缺字段请求
- 非法字段请求
- 重复事件请求
- 媒体字段为空时的请求
- 旧界面与新 Web 并存时的请求路径

---

## 10. 结论

兼容接口不是临时工作，而是本项目第一阶段的核心保护面。

在真正完成独立后端和新 Web 前端之前，以下要求必须始终成立：

- 旧接口尽量稳定
- 上游调用尽量不变
- 控制器不再继续耦合 UI
- 内部重构优先通过应用服务和适配层完成
