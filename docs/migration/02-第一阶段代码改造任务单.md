# Uixe.Watcher 第一阶段代码改造任务单

## 1. 文档目的

本文档将第 1 阶段“后端解耦”拆解为可直接执行的代码改造任务，用于指导实际开发顺序、评审范围和验收标准。

本阶段核心目标：

- 拆掉 `Controller -> Form` 直接调用
- 建立应用服务边界
- 保持旧接口可用
- 保持旧界面可运行
- 为独立后端项目做准备

---

## 2. 本阶段必须遵守的规则

- 不直接删除现有 `WinForms` 内容
- 现有界面与新 Web 界面允许并存，并可切换
- 不新增任何新的 `Controller -> Form` 直接依赖
- 不先破坏上游接口契约
- 先增加适配层，再迁移实现
- 所有新增核心逻辑优先落在应用服务边界内

---

## 4. 任务清单

### 任务 1：梳理 `Program.cs` 现状

目标：

- 明确当前宿主启动方式
- 明确 `WinForms` 与 `ASP.NET Core` 的耦合点
- 标记未来独立后端可迁移的启动逻辑

产出：

- 启动链路说明
- 宿主职责拆分建议

验收：

- 能说明哪些逻辑应留在当前宿主
- 能说明哪些逻辑应迁移到独立 `Api`

### 任务 2：梳理 `Startup.cs` 服务注册

目标：

- 明确当前依赖注入注册内容
- 识别 UI 依赖、配置依赖、服务依赖
- 划分哪些注册未来应进入 `Api`、`Application`、`Infrastructure`

产出：

- 服务注册清单
- 迁移目标清单

验收：

- 能清楚标记 UI 相关注册与业务相关注册

### 任务 3：改造 `LaneController`

目标：

- 去掉对 `frmPlaza` 的直接访问
- 保持现有接口对外契约不变
- 改为调用应用服务接口

建议步骤：

1. 提取 `ILaneApplicationService` 或等价接口
2. 将控制器中的业务逻辑迁移到应用服务
3. 保留控制器中的参数接收与返回映射
4. 将 UI 刷新需求转为标准结果或事件输出

验收：

- `LaneController` 不再直接持有窗体依赖
- 旧接口仍能被调用

### 任务 4：改造 `TrafficEventQueueService`

目标：

- 去掉对 `frmPlaza` 或其他窗体类型的依赖
- 拆成更明确的职责边界

建议拆分方向：

- `EventIngestionService`
- `LaneMatchingService`
- `MediaResolverService`
- `RealtimePushService`
- `NotificationService`

验收：

- 事件处理流程不再直接驱动窗体
- 队列处理逻辑与展示逻辑分离

### 任务 5：下沉窗体中的业务逻辑

目标：

- 梳理 `frmMain`、`frmPlaza` 中承载的业务逻辑
- 把可迁移部分转移到应用服务或辅助服务

优先迁移：

- 车道匹配
- 站点定位
- 事件整理
- 媒体整理
- 去重判断
- 推送通知前的数据准备

验收：

- 窗体更多只负责展示和交互
- 业务逻辑不再继续集中堆积在窗体层

### 任务 6：建立兼容适配层

目标：

- 在旧接口与新应用服务之间建立适配边界
- 不破坏上游联调方式

内容：

- 请求映射
- DTO 适配
- 返回结果映射
- 兼容日志记录

验收：

- 新旧实现切换时，上游感知尽量最小

### 任务 7：定义最小应用服务接口集

建议至少定义：

- `IEventIngestionService`
- `ILaneMatchingService`
- `IRealtimePushService`
- `INotificationService`
- `IMediaResolverService`
- `ISystemConfigService`

验收：

- 首批核心流程可通过接口组织
- 不再需要控制器直接访问窗体实现核心逻辑

---

## 5. 推荐执行顺序

推荐按以下顺序推进：

1. 盘点 `Program.cs` 和 `Startup.cs`
2. 梳理 `LaneController` 现状
3. 梳理 `TrafficEventQueueService` 现状
4. 提取应用服务接口
5. 改造 `LaneController`
6. 改造 `TrafficEventQueueService`
7. 下沉 `frmMain` / `frmPlaza` 中的业务逻辑
8. 增加兼容适配层和回归验证

---

## 6. 风险点

### 6.1 主要风险

- 业务逻辑散落在窗体事件中，抽离成本高
- 老接口缺少完整文档，容易误改
- 旧界面刷新逻辑与业务处理强耦合
- 新旧实现并存期间状态同步复杂

### 6.2 控制手段

- 先梳理再改代码
- 每次只抽一条业务链路
- 控制器只做薄层改造
- 窗体逻辑不一次性大迁移
- 关键接口保留回归样例

---

## 7. 验收标准

本阶段完成时，至少满足以下条件：

- `LaneController` 不再直接操作 `frmPlaza`
- `TrafficEventQueueService` 不再直接驱动窗体展示逻辑
- 旧接口仍可访问
- 旧界面仍能运行
- 新增应用服务接口已形成雏形
- 新增代码没有继续强化 UI 耦合

---

## 8. 暂不做事项

本阶段明确不做：

- 一次性删除旧 `WinForms`
- 一次性把全部业务迁完
- 强制上游切换到新接口
- 先全面重做前端
- 先全面替换数据库

---

## 9. 阶段结论

第一阶段不是“重写”，而是“拆耦合 + 建边界 + 保兼容”。

只要始终满足以下三条，本阶段就是正确推进：

1. 旧接口还在
2. 旧界面还能跑
3. 新业务边界已经开始形成
