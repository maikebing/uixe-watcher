# Uixe.Watcher 分阶段迁移清单

## 1. 文档目的

本文档用于约束 `Uixe.Watcher` 的迁移顺序、交付物、风险控制点和回滚策略，确保项目按“先兼容、后替换、逐步解耦”的方式推进。

适用范围：

- 现有 `WinForms` 主程序
- 现有进程内 `ASP.NET Core`
- 后续独立 `Web API`
- 后续 `Web` 前端
- 可选 `Agent` 兼容层

---

## 2. 总体迁移原则

### 2.1 基本原则

- 不一次性重写全部系统
- 不直接删除现有内容
- 不先删除 `WinForms`
- 不先要求上游系统改接口
- 现有界面与新 Web 界面允许并存，并可切换
- 先稳定接口，再逐步迁移实现
- 先抽离业务，再替换展示

### 2.2 迁移总目标

从当前模式：

`外部系统 -> 本机内嵌 API -> WinForms 窗体`

迁移到目标模式：

`外部系统 -> 中心后端 API -> Application Service -> 数据存储/缓存 -> SignalR -> Web 前端`

可选补充：

`中心后端 -> Agent -> 本地弹窗/语音/VNC`

---

## 3. 迁移阶段总览

| 阶段 | 目标 | 预计周期 | 核心输出 |
| --- | --- | --- | --- |
| 第 0 阶段 | 梳理与冻结 | 2~3 天 | 接口、DTO、依赖清单 |
| 第 1 阶段 | 后端解耦 | 1~2 周 | 应用层边界、独立后端骨架 |
| 第 2 阶段 | 实时与存储 | 1 周 | SignalR、入库、缓存、历史模型 |
| 第 3 阶段 | Web 前端一期 | 2~3 周 | 监控端基础页面 |
| 第 4 阶段 | Agent 兼容层 | 1~2 周 | 本地兼容与能力迁移 |
| 第 5 阶段 | 联调与灰度 | 1 周 | 灰度上线版本 |

---

## 4. 第 0 阶段：梳理与冻结

### 4.1 阶段目标

在不改业务行为的前提下，梳理当前系统真实边界，形成迁移基线。

### 4.2 必做事项

- 盘点全部 HTTP 接口
- 盘点全部核心 DTO
- 梳理 `Controller -> Form` 直接依赖点
- 梳理 `TrafficEventQueueService` 所在事件链路
- 梳理本地能力清单
- 梳理上游系统对接方式
- 明确必须保留的返回结构与行为

### 4.3 重点扫描文件

- `Program.cs`
- `Startup.cs`
- `Controllers/LaneController.cs`
- `Services/TrafficEventQueueService.cs`
- `WinForms/frmMain.cs`
- `WinForms/frmPlaza.cs`

### 4.4 交付物

- 现有接口清单
- DTO 清单
- 页面清单
- 本地能力清单
- 上游对接清单
- 风险点清单

### 4.5 阶段完成标准

满足以下条件才可进入下一阶段：

- 关键接口已列全
- 核心 DTO 已梳理完成
- 主要耦合点已定位
- 不再依赖口头认知推进开发

---

## 5. 第 1 阶段：后端解耦

### 5.1 阶段目标

建立未来独立后端的最小骨架，并把 `Controller` 与 UI 的直接耦合拆开。

### 5.2 必做事项

- 新建分层项目边界
- 建立 `Api`、`Application`、`Domain`、`Infrastructure`、`Contracts` 的职责划分
- 将 `Controller -> Form` 调用改为 `Controller -> Application Service`
- 提炼共享 DTO 与结果模型
- 抽离事件处理主链路的服务接口

### 5.3 第一批优先改造对象

1. `Program.cs`
2. `Startup.cs`
3. `Controllers/LaneController.cs`
4. `Services/TrafficEventQueueService.cs`
5. `frmMain.cs`
6. `frmPlaza.cs`

### 5.4 建议新增核心服务

- `IEventIngestionService`
- `ILaneMatchingService`
- `IMediaResolverService`
- `IRealtimePushService`
- `INotificationService`
- `ISystemConfigService`

### 5.5 交付物

- 独立后端项目骨架
- 应用服务接口清单
- 兼容层映射方案
- 初版依赖注入边界

### 5.6 阶段完成标准

- `Controller` 不再直接操作窗体
- 核心业务逻辑开始迁移到应用服务
- 旧接口仍可继续访问
- 旧界面仍可运行

---

## 6. 第 2 阶段：实时通信与数据落库

### 6.1 阶段目标

让后端具备独立接收请求、事件入库、状态缓存和实时推送能力。

### 6.2 必做事项

- 引入 `SignalR`
- 建立事件入库模型
- 建立站点与车道状态缓存
- 梳理历史查询读模型
- 补充实时推送消息契约

### 6.3 交付物

- 事件持久化能力
- 状态缓存能力
- SignalR 推送能力
- 历史查询基础模型

### 6.4 阶段完成标准

- 后端可以脱离窗体完成主流程处理
- 关键事件可查询、可回溯、可推送
- Web 前端具备接入基础

---

## 7. 第 3 阶段：Web 前端一期

### 7.1 阶段目标

构建第一版监控 Web 前端，承接未来主要展示功能。

### 7.2 一期页面范围

- 监控总览
- 收费站监控
- 事件中心
- 事件详情
- 历史查询
- 系统配置

### 7.3 必做事项

- 建立前端工程骨架
- 对接后端 API
- 对接 `SignalR`
- 建立布局、路由、状态管理
- 建立基础深色主题和卡片体系

### 7.4 交付物

- 可运行的 Web 前端一期
- 与后端联通的实时页面
- 页面切换与基础权限框架

### 7.5 阶段完成标准

- 新 Web 界面能承接核心监控浏览能力
- 用户可在旧界面和新界面间并行使用

---

## 8. 第 4 阶段：Agent 兼容层

### 8.1 阶段目标

在需要保留本地能力的前提下，通过 `Agent` 承接本地能力和旧接口兼容。

### 8.2 Agent 职责

- 监听旧接口
- 转发到中心后端
- 执行本地通知
- 执行本地语音播报
- 调起 `VNC`
- 承载 `WebView`

### 8.3 必做事项

- 实现本地兼容转发
- 迁移语音提醒
- 迁移本地通知
- 迁移 `VNC` 能力
- 增加登录后界面切换入口

### 8.4 交付物

- 可运行的 `Agent`
- 本地能力兼容清单
- 新旧界面切换方案

### 8.5 阶段完成标准

- 本地能力不依赖旧主业务流程
- 旧客户端具备向 `Agent` 过渡的条件

---

## 9. 第 5 阶段：联调与灰度

### 9.1 阶段目标

在真实环境验证新架构可用性，并控制上线风险。

### 9.2 必做事项

- 与上游系统联调
- 多站点试运行
- 压力测试
- 兼容性验证
- 准备回滚预案

### 9.3 交付物

- 灰度上线版本
- 联调问题清单
- 压测结果
- 回滚说明

### 9.4 阶段完成标准

- 关键业务链路稳定
- 上游兼容通过
- 回滚路径清晰可执行

---

## 10. 迁移中的兼容策略

### 10.1 优先兼容接口

第一阶段优先保障以下入口：

- `LaneController`
- `TrafficEvent([FromBody] TrafficEventPushRequest)`
- `Bulktrans(string, BulklyDto)`
- `bill_info(string, BillInfoDto)`
- `ConfirmEnInfo`

### 10.2 兼容原则

- 入参结构尽量不变
- 返回结构尽量不变
- 内部允许新增适配层
- 不对上游提出先改造要求

### 10.3 展示兼容原则

- 旧界面保留
- 新 Web 界面逐步接管展示
- 两套界面允许并行使用
- 允许通过登录或配置切换入口

---

## 11. 每阶段都必须执行的检查项

### 11.1 开发前

- 是否明确本阶段边界
- 是否明确不能动的旧接口
- 是否明确回滚方式

### 11.2 开发中

- 是否出现新的 `Controller -> Form` 直接依赖
- 是否把 UI 逻辑继续写回业务层
- 是否破坏旧返回结构

### 11.3 开发后

- 是否完成最小联调验证
- 是否保留旧界面运行能力
- 是否记录新增风险与技术债

---

## 12. 回滚策略

### 12.1 回滚原则

- 每阶段都必须可独立回退
- 不把新旧逻辑一次性绑死
- 关键切换点必须配置化或开关化

### 12.2 推荐回滚方式

- 保留旧接口实现路径
- 保留旧界面入口
- 新功能通过配置开关控制
- 新旧推送路径可切换
- 新存储接入前保留旧运行链路

---

## 13. 迁移执行结论

本项目迁移的正确方式不是一次性替换，而是：

- 先梳理
- 再解耦
- 再独立后端
- 再接 Web
- 再补 Agent
- 最后联调和灰度

整个过程中，必须始终满足三个条件：

1. 旧接口尽量稳定
2. 旧界面可继续运行
3. 新 Web 界面可逐步并存切换
